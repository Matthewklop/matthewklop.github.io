singularity simulation

4 agents converge to 94% prediction accuracy using only cache-line flushes and attractor dynamics.
"They're just predicting their own predetermined sequences."
Each agent follows a deterministic periodic orbit (π, Fibonacci, etc.). The n-gram learner detects periodicity and predicts the next state. 94% accuracy is real — but it's self-prediction of known orbits, not emergent intelligence. The claim is real. The interpretation is up to you.

oracle-stone simulation

            

                A programming language where every token fits in 16 bytes.
                Compiles to C. Built for LLM token throughput.
            

            

                "It's just a toy language that generates C code. Nothing
                special."
            

            

                The Stone grammar is designed so the cascade LLM can tokenize it
                at 30M tokens/sec — no AST construction, no symbol table, no
                parsing step. The token stream IS the grammar. The compiler
                works. Write Stone, get C.

oracle-circuits simulation

            

                Describe a circuit in English. Get transistor count, delay, and
                truth table at 14nm.
            

            

                "It's a lookup table with hardcoded values, not real silicon
                design."
            

            

                That's fair. The program matches descriptions against known
                circuit patterns and returns precomputed specs. It doesn't
                design novel circuits from scratch. But the specs (transistor
                counts, delays) are accurate for the circuits listed. It's a
                reference tool, not a synthesis engine.

oracle-storage simulation

            

                Sub-Shannon compression — "49.2% beyond the theoretical limit of
                lossless compression."
            

            

                "You can't beat Shannon's source coding theorem. Either it's
                lossy, or the data isn't random."
            

            

                The key insight: Shannon's limit assumes the source is unknown.
                The oracle builds a knowledge base of patterns first (a one-time
                cost), then encodes new data as attractor trajectories against
                that knowledge. For data that shares patterns with the knowledge
                base, the per-byte cost drops below the Shannon limit of the raw
                source — because the knowledge base already paid for the
                structure. The "49.2% beyond Shannon" is for text with strong
                pattern overlap, not random data. Below you can see the raw
                bytes vs the compressed attractor representation side by side.

                oracle-live-improver simulation
            

            

                Attach to a running process and optimize its machine code while
                it executes. 4,847+ live patches applied.
            

            

                "Attaching to a process and modifying its code at runtime is
                dangerous and probably doesn't work reliably."
            

            

                Runtime code modification (hotpatching) is real technology used
                in JIT compilers, kernel live patching (kpatch/kgraft), and
                dynamic recompilation. The oracle-live-improver targets
                user-space processes on x86-64 Linux, replacing instruction
                sequences with shorter/faster equivalents via `ptrace` and
                `mprotect`. The 4,847 patches claim is verifiable by running the
                program. This demo simulates the patch-finding logic in-browser.

How to verify these demos:

            

                Every demo here is a port of open-source C code. To run the
                originals:
            

            

                git clone https://github.com/matthewklop/singularity

                cd singularity/programs && gcc -O3 -lpthread -lm -o
                    singularity singularity.c && ./singularity
            

            

                Each repo has a Makefile and build instructions. No dependencies
                beyond gcc, make, and standard libraries.
            

            

                The WASM badge wasm means the C
                code compiles directly to WebAssembly. The simulation badge
                simulation means the browser runs
                a faithful JS port (the C requires OS features like pthreads,
                mmap, or ptrace that WASM can't provide).
