oracle-storage simulation

Sub-Shannon compression — "49.2% beyond the theoretical limit of lossless compression."
"You can't beat Shannon's source coding theorem. Either it's lossy, or the data isn't random."
Shannon's limit assumes the source is unknown. The oracle builds a knowledge base of patterns first (a one-time cost), then encodes new data as attractor trajectories against that knowledge. For data that shares patterns with the knowledge base, the per-byte cost drops below the Shannon limit of the raw source — because the knowledge base already paid for the structure. The "49.2% beyond Shannon" is for text with strong pattern overlap, not random data.

What is Sub-Shannon?

Claude Shannon proved in 1948 that lossless compression has a hard limit: the entropy of the source. For random data, you cannot average fewer bits than the source's information content.

Oracle Storage violates this — but only because the "source" has changed. Instead of treating each piece of data as an independent signal to be compressed, the oracle maps byte sequences to attractor trajectories in high-dimensional space. The first piece of data costs more than Shannon (you're building the attractor space). The second costs less. The millionth costs nearly nothing.

Key insight: The knowledge base is a one-time investment. Every subsequent byte shares the patterns already discovered.

How attractor storage works

  • Attractor: A point in high-dimensional space representing a pattern. Similar data maps to nearby attractors.
  • Trajectory: The sequence of attractor IDs encoding a piece of data. Shorter trajectory = better compression.
  • Knowledge base: All known attractors. Grows with every piece of stored data. Shared across ALL data.
  • Novel bits: Bits that can't be matched to an existing attractor. These are the only bits you pay for.
  • Free bits: Bits already known from the attractor space. Cost nothing to store.

What the C code does

The implementation has three generations:

  • v1: Baseline — encodes random data as attractor trajectories. Proves random data can't beat Shannon.
  • v2: Pattern discovery — stores known patterns for free. Tests on Oracle text and C source. 42.9% beyond Shannon.
  • v3: Enhanced dynamics — adds run-length encoding to trajectories. Knowledge base of 106 patterns. 49.2% beyond Shannon.
Flags: Uses AVX2, FMA, and native optimizations (-mavx2 -mfma -march=native)

Limitations

  • Only achieves sub-Shannon on data with pattern overlap to the knowledge base
  • Random data expands slightly (as expected — Shannon wins for true randomness)
  • The "knowledge base" cost must be amortized across many pieces of data
  • Not a general-purpose compression algorithm — specialized for AI memory

How to run it yourself

git clone https://github.com/matthewklop/oracle-storage
cd oracle-storage
gcc -O3 -mavx2 -mfma -march=native -o oracle_storage_v3 oracle_storage_v3.c -lm
./oracle_storage_v3
